# Spec: Vet Weekly Schedules

## Goal

Allow clinic staff to configure weekly working schedules for each veterinarian. Each vet gets a per-day schedule (which days they work, start time, end time). Vets without a configured schedule will not appear as available for booking — but the booking flow integration itself is a separate task.

## Acceptance Criteria

1. Staff can view each vet's configured working days and hours.
2. Staff can add, edit, and delete schedule entries for any vet.
3. A schedule entry defines a day of the week with start and end working hours.
4. A vet has at most one schedule entry per day of the week.
5. Invalid entries — duplicate days or end time not after start time — are rejected with clear feedback.
6. Schedule management is reachable from the main navigation.
7. Seed data covers varied schedules including at least one vet with no schedule configured.

## Data Model
One vet → many schedule entries (up to 7, one per day). The `day_of_week` integer uses ISO-8601 numbering (Monday=1 through Sunday=7) to align with Java's `DayOfWeek.getValue()`.

## Implementation Notes
**No existing CRUD pattern to follow.** The project has create/edit forms for owners and pets, but no page with a table of entries plus inline add/edit/delete. The closest analog is the owner details page (table of pets/visits), but it doesn't support editing rows. The schedule management page will be the first true CRUD table in the app.

**Feature-based packaging.** The `VetSchedule` entity, repository, controller, and validator all go in the `vet` package (`org.springframework.samples.petclinic.vet`), alongside `Vet.java` and `VetRepository.java`.

**Snake-case naming strategy.** The project uses `PhysicalNamingStrategySnakeCaseImpl`, so Java field `dayOfWeek` automatically maps to column `day_of_week`. Schema files still use explicit snake_case column names.

**`DayOfWeek` storage.** Store as integer (1–7) rather than a string enum. This keeps the schema simple, sorts naturally, and maps directly to `java.time.DayOfWeek.getValue()`. The template displays the localized day name via the entity's convenience method.

**No authentication system.** The app has no user roles or auth. The schedule management page is accessible to anyone, same as all other pages. The PRD says "configured by clinic staff" but role-based access is out of scope.

**VetRepository changes.** The existing `VetRepository` extends `Repository` (minimal interface) and only exposes `findAll()`. The schedule controller needs `findById()`, so either: (a) add `findById` to `VetRepository`, or (b) inject `VetScheduleRepository` and navigate to the vet through the schedule. Option (a) is cleaner — add `Optional<Vet> findById(Integer id)` to `VetRepository`.

**Cache considerations.** `VetRepository.findAll()` is `@Cacheable("vets")`. The new `findById` method should also be `@Transactional(readOnly = true)` but does not need caching (it's only used in the schedule management flow, not a hot path).

## Decisions

### One entry per day per vet (not multiple time blocks)
A vet's schedule for a given day is a single continuous block (e.g., 09:00–17:00). This matches the PRD language ("which days they work and their working hours") and avoids the complexity of merging/splitting overlapping time ranges. If split shifts are needed later, the unique constraint changes from `(vet_id, day_of_week)` to a range-overlap check — a deliberate future extension, not an oversight.

### POST for delete (no DELETE method)
HTML forms only support GET and POST. Using `POST /vets/{vetId}/schedules/{id}/delete` avoids requiring JavaScript or Spring's `HiddenHttpMethodFilter`. This matches the project's server-rendered, no-JS approach.

### Schedule management is a separate page, not inline on vet list
The vet list page (`/vets.html`) is a read-only, paginated directory. Embedding schedule CRUD there would bloat the page and mix concerns. A dedicated `/vets/{vetId}/schedules` page keeps the vet list clean and gives schedule management room to grow.

## Out of Scope

- **Booking flow integration** — filtering vets by schedule availability is a separate task.
- **Role-based access** — no authentication or staff-only restrictions on the schedule page.
- **Split shifts / multiple blocks per day** — one continuous working block per day.
- **Schedule exceptions / overrides** — no way to mark a vet as unavailable on a specific date (e.g., holidays, sick days). Weekly schedule only.
- **Schedule history / audit log** — changes are not tracked.
- **Vet self-service** — vets cannot manage their own schedules (no vet login).
